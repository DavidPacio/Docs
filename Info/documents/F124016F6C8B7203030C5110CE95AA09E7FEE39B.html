In computer science and information science, an ontology encompasses a representation, formal naming, and definition of the categories, properties, and relations between the concepts, data, and entities that substantiate one, many, or all domains.
 
Every field creates ontologies to limit complexity and organize information into data and knowledge. As new ontologies are made, their use hopefully improves problem solving within that domain. Translating research papers within every field is a problem made easier when experts from different countries maintain a controlled vocabulary of jargon between each of their languages.[1]

I do plan to use many of the Braiins SIM ideas in Pacio's SSIM, but extended, and made more flexible still in three main ways:
 
  • extending properties to be semantic triples as per WC3's RDF. See https:⁄⁄en.wikipedia.org⁄wiki⁄Semantic_triple
(XBRL properties and the Braiins equivalent are "is" descriptors only e.g. this is a rent expense. SSIM ones will allow any relationship to be defined, plus with other info optionally attached e.g. 25% owned by xxxx)
 
  • making what we currently call SDOs (Smart data Objects) even smarter than they were for the Braiins work where they were called Bros (Braiins Report Objects)
 
  • extending everything to be more general and to work with blockchain⁄decentralised data
 
The pieces involved are:
 
  • Taxonomies which define how data is organised. There will a lot of these. Our taxonomies will allow for much more than XBRL taxonomies do, but it will be possible to import XBRL taxonomies to produce a Pacio equivalent, using a subset of our Taxonomy capability. Our taxonomies will also be able to import Schema.org schemas. For the Braiins work I developed code to import XBRL taxonomies into MySQL database tables. The taxonomies were then MUCH better to work with than when they were in their native XML format. I will do this again, but allow for much more. Our library of Taxonomies will become a good source of info in their own right. Maintaining them will be one of the things that PCL will do, under a nominally different name akin to the original idea of having a separate Pacio Foundation for this sort of stuff. I just checked pacio.org but that is not available though it is not in use. Pacio.app is available though!
 
  • SDOs which are described by and work with:
    * folio which defines allowable properties
      - properties (dynamic unlike XBRL ones e.g. Director.MaryLou, and triples e.g. has a third party relationship with xxxx) with details via:
         - members with attributes for the member [many]
    * attributes of the SDO [many]
    * SDO code
    (There is a lot involved with SDOs. They are what will define SSIM and Pacio.)
 
  • SdoSets which are groupings of SDOs according to a particular reporting requirement. An SDO can be used in multiple SdoSets. SdoSets also have optional properties⁄attributes re:
     * taxonomy or taxonomies applicable
     * countries⁄jurisdictions⁄standards⁄language etc applicable
 
  • Journals for posting in an accounting app where Drs must equal Crs, and other restrictions according to the Taxonomy etc apply. Journals would involve an SdoSet. Non-accounting structured data would involve a create or edit transaction as for BigchainDB, but not a 'journal'. Schedule single entry type data for an accounting app, which does not need to balance, would still use journals for organisational⁄reference⁄audit trail purposes, but not enforce a balancing requirement. It would be up to an app as to whether a journal must be one or the other type of posting or could allow a mixture.
 
  • Digital Id for which we will have a complete subsystem that needs to work with the many other attempts out there. One interesting find during the week was this:
     Decentralized Identity Trilemma https:⁄⁄decentralized.id⁄
 
  • A set of services or subsystems to handle:
     * digital Id
     * taxonomy stuff
     * app directory (cross app) stuff
     * SDO handling i.e. to allow use of SDOs for posting⁄querying⁄reporting
     * context stuff
     * reference stuff
     * posting (or whatever we call it, that being an old accounting term)
     * real time inputs re IoT, AI interfaces, and Tari
     * data feeds (Oracle stuff)
     * data exports subsystem incl dynamically for use by the next subsystem:
     * an opt-in cross anything⁄everything query⁄analysis⁄reporting system using homomorphic encryption to preserve privacy
     * interfaces to other protocols XBRL, RDF, UBL, Open EDI etc
 
 
You can gain some appreciation of these pieces, except for the Digital Id, triple stuff and the subsystems, via the docs mentioned in my previous "More SSIM Meat" email, content repeated below.
 
So, we have attributes in 3 or possibly 4 places
 
And SDO is not pronounceable. Bro was. But I want to avoid the "R" of Report this time, as these objects are not just for reporting. They are for posting, storage, and querying too.
 
Use just DO for data Objects instead? Then we would have DOs or Dos (memories of a certain awful operating system), DoSets, Master DO, Slave DO etc.
 
Or DOE for Data Object Extended. That would avoid the DOS acronym, giving us Does, DoeSets, Master Doe, Slave Doe ...
