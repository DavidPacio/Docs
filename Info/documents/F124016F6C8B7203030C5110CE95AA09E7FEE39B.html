The Pacio Plan (A work in progress)
 
Narrower Focus for Pacio
Proposed decision: Initially focus on Business Financial and <IR> Data to make the task more manageable, and to better fall within the realm of our knowledge⁄experience, while keeping "all data" in mind as a long term objective. The initial focus would include Tari data.
 
The need to narrow Pacio's initial approach was clarified by:
  • the difficulty people have had in grasping what Pacio is all about, and our difficulty in explaining it
  • the research for this work which has shown just how much "semantic web" effort has been expended, with no clear winner emerging
  • articles such as this one What to do with the data? The evolution of data platforms in a post big data world which posits that "We are in a post-big data world" and state that "What we did end up with is lots and lots of data in silos, isolated from each other, overwhelming the users that are trying to figure out how to use it, and the IT people who are trying to figure out how to manage it." = the leap for Pacio to go to semantic data for everything initially is too great to be achievable or realisable with our current or likely resources. See the Web 3 - Semantic Data node for more articles.
 
This decision will involve simplifying the White Paper and the web site, plus possibly coming up with a new tag line instead of "Data without Boundaries." The current white paper could be split into two to cover the near term and the MVP, plus the long term plan.
 
The rest of this "plan" assumes the above.
 
Requirements for Pacio Semantic Data System and Application Platform built to give Apps Access to it
Natural and easy for end users - business people, investors, regulators, individuals, anybody - to use without study when
* entering⁄creating⁄filing data
*   querying⁄searching data
*   creating reports, whether they be Financial Statements, so called <IR> reports, or just ad hoc reports
 
Achieved by
* use of tags⁄key words in natural languages which
  - handle misspellings
  - handle homograph, homophone, homonym, and heteronym words
  - handle associations⁄related thoughts⁄terms
* prompting from context⁄meaning of the data in question⁄what has been entered
* supporting voice input
* avoiding the W3C RDF type triples with urls and other such complicated⁄ugly attempts
 
With this being made to working behind the scenes via key components of:
 
Ontologies
Ontologies will define how data is structured, organised, and related, using SOSOs (Semantic Ontology Smart Objects) to handle the complexities, without users (other than ontology creators⁄maintainers) needing to know that SOSOs even exist.
 
Ontologies will encompass:
* W3C type ontologies
* schemas as in Schema.org
* taxonomies as in XBRL taxonomies
* any other required "ontology" as per Ontologies Ontologies Everywhere – but Who Knows What to Think? for which an import can be written
 
Ontologies will include Braiins SIM like features (references below):
* folios which defines allowable properties
  - properties with details via:
    . members with attributes for the member [many]
* attributes of the SOSO [many]
* SOSO code
 
which provide properties with order, hierarchy, and relationships as per structured (linked data) ontologies
 
while also allowing for properties without order or hierarchy as per tags in the article Goodbye, Object Oriented Programming
and its conclusion for the way to overcome problems with inheritance in OO languages of using tags:
Tags have no order or hierarchy. (This solves the Diamond Problem too.)
Tags are analogous to interfaces since you can have multiple types associated with the document.
The author's tags are also analogous to our properties, properties being a tag plus attributes and members.
 
SOSO properties will be dynamic unlike XBRL ones e.g. Director.MaryLou, and support semantic triple type relationships as per WC3's RDF but done using test based tags and relationship operators without complex url⁄XML schemes.
 
Ontology "truth" re structure and relationships will be achieved by a consensus methodology to be developed by Pacio - a wikipedia like open source world wide group of interested people but with better consensus courtesy of blockchain to avoid trolling and barrow pushing. Inspired in part by the FOAM Project and the Noam Chomsky Knowledge Graph project. Must allow for updating though. Cannot be immutable.
 
Ontologies will allow importing of existing ontologies, relevant knowledge graphs, and taxonomies especially XBRL ones.
 
Ontologies will be stored in a "Global DB". DB to be decided. BigchainDB or TiDB⁄TiKV as per ont.io
 
There will be many ontologies, ultimately thousands of them for all types financial reporting requirements.
 
These smart object⁄DB based ontologies will be much more powerful than other methods of expressing semantic data relationships such as W3C's OWL ontologies or XBRL Taxonomies yet will be easier to work with and understand.
 
Developing and maintaining the ontologies will be one of the things that PCL will do, under a nominally different name akin to the original idea of having a separate Pacio Foundation for this sort of stuff. Pacio.org is not available though it is not in use. Pacio.app is available though.
 
Digital Identity System:
* global, using same DB as the Ontologies
* open with interfaces to all the common Digital ID systems as per Digital Identity and the Digital Id section of the White Paper
* one challenge to be overcome is the Decentralized Identity Trilemma
 
SSIM (Standardised Semantic Information Model) data storage system:
*  uses DOEs (Data Objects Extended) or Does, previously called SDOs (Smart Data Objects)
*  works with:
  - SOSOs
  - digital Ids
  - a flexible mixture of public and private blockchain⁄decentralised data storage systems including:
    . BigChainDB
    . Holochain
    . IPFS
    . Radix?
    . Others?
  - DoeSets which are groupings of Does including Master and Slave Does, according to a particular reporting requirement. A Doe can be used in multiple DoeSets. DoeSets also have optional properties⁄attributes re:
     . ontology applicable and thus SOSOs
     . countries⁄jurisdictions⁄standards⁄language etc applicable
  - Journals for posting in an accounting app where Drs must equal Crs, and other restrictions according to the Ontology⁄SOSOs apply. Journals would involve a DoeSet. Non-accounting structured data would involve creating or editing a transaction, as for a BigchainDB transaction, but not a 'journal'. Schedule single entry type data for an accounting app, which does not need to balance, would still use journals for organisational⁄reference⁄audit trail purposes, but not enforce a balancing requirement. It would be up to an app as to whether a journal must be one or the other type of posting or could allow a mixture.
 
A set of services or subsystems to handle:
* digital Id
* app directory (cross app) stuff
* ontology stuff incl SOSO handling
* Doe handling i.e. to allow use of DOEs for posting⁄querying⁄reporting
* context stuff
* reference stuff
* posting (or whatever we call it, that being an old accounting term)
* real time inputs re IoT, AI interfaces, and Tari
* data feeds (Oracle stuff)
* data exports subsystem incl dynamically for use by the next subsystem:
* an opt-in cross anything⁄everything query⁄analysis⁄reporting system using homomorphic encryption to preserve privacy
* interfaces to other protocols XBRL, RDF, UBL, Open EDI etc
