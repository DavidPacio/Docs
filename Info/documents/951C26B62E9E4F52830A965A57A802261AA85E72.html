The 'Pacio Storage.txt' ideas
 
These 2017 notes assume that the Pacio blockchain allows customisation of the blockchain (BC or bc) transaction⁄message.
Note: Probably NOT how it will be done in 2018++, but kept just for reference⁄ideas.
 
Bc transaction to have a Pacio field added at the end. Format of that i.e. delimited, specified length etc to be decided when the transaction detail is available.
 
These notes assume English only, but support for other languages will need to be added.
 
BC Transaction (Message) Pacio Field
====================================
- transaction type Id #
- app Id
- entity Id
- fields as defined for transaction type, in binary form
 
Database About Pacio Transactions for use by Producers Building Blocks and Apps via the API to be able to pass appropriate Json objects
=======================================================================================================================================
After live running starts the records described here which include a version # cannot be edited.
Change must be made via a new entry with a link back to the previous version.
 
Questions?
- dispense with enums and use ids for everything for consistency?
 
Constants
~~~~~~~~~
Enums for:
- app type:          app, mapp, dapp, chatbot, ...
- continents:        Africa, America North, America South, Asia, Australia, Asia, Europe
- country groupings: monetary union, customs union, political union, trade association, military alliance, multi-functional union
 
Tables
~~~~~~
Transaction Types In memory DB⁄table
-----------------
Id ->
- class      alpha 32 chars
- subClass   alpha 32 chars
- sortKey    alpha 32 chars
- ver #      int with 2 places of implied decimals re fields changes
- prevId     id of previous version after an edit
- fields     list of fields where each field descriptor holds info about the field, its type, and its purpose, in json form
- creation   datetime
- edit       datetime
 
Notes: Every single write action in Pacio, including via the API, will involve a transaction, so there will be a lot of transactions,
       as well as a lot of transaction types to describe them.
 
Apps
----
Id ->
- app type   enum
- name       alpha 32
- short Name alpha 16
- ver #      int with 2 places of implied decimals
- prevId     id of previous version after an edit
 
World Regions
-------------
Id ->
- name               alpha:     South East Asia
- short Name if any  alpha:     SEA
- continent(s)       enum list: Asia Enum
 
Country Groupings
-----------------
Id ->
- type               enum:  multi-functional union
- name               alpha: European Union
- short name if any  alpha: EU
 
Countries
---------
Id ->
- name             alpha
- ISO name         alpha
- propertyName     no spaces form
- short name       if different from name
- ISO-31661-1      #
- part of          country id  e.g. England part of UK
- associated with  country id  e.g. Gibraltar associated with UK
- continent        enum
- region(s)        id list
- grouping(s)      id list
- comments         alpha
 
Entity Types
------------
Id ->
- name             alpha
- ver #            int with 2 places of implied decimals re fields changes
- prevId           id of previous version after an edit
- parentEntityId   0 if none, otherwise properties merged
- countries        id list, empty if not country specific e.g. 'Individual' or DAO
- groupings        id list, empty if none
- fields           list of fields where each field descriptor holds about the field, its type, and its purpose, in json form
- comments         alpha
 
Entities
--------
Id ->
- entity type      id
- accountAddr      SEA BC address
- fields as defined for the entity type, in binary form
 
And so on for:
 
- Currencies (Crypto and Fiat)
 
- Languages
 
- Stock Exchanges
 
- Crypto Exchanges
